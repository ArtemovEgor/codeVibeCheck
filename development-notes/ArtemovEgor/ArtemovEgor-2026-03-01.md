# 25 февраля 2026 — Связываю API авторизации с формами

За эту неделю, как и планировал, закончил API сервис, API слой для авторизации и UI авторизации и регистрации. Делал это отдельно в двух разных ветках, и ближе к концу столкнулся со сложностями, когда нужно было связать UI и API из-за того, что, что измнения нужно было внести в разные компоненты (которые работают с авторизацией), и часть из них была релизнута или измнена уже когда я работал над API. Параллельно другая участница добавляла layout главной страницы, которому нужно знать имя пользователя и пользоваться методом logout, и также были внесены измнения в роутер, в который и я вносил измнения, поэтому дополнительно несколько часов потратил на то, чтобы разрешить все конфликты и связать все. В итоге все работает, и, надеюсь, можно приступить уже к core-функционалу.

При разработке API-сервиса, сразу решил сделать разделение на реальный и mock-backend. В основном пришлось подумать над разделением ответственности компонентов: должен ли моковый API авторизации знать о реальном, чтобы с его помощью чистить токены (решил, что нет), должна ли быть единая точка управления моковыми запросами, или под каждый API сервис нужно писать свой (тут пока пришел ко второму варианту, чтобы не сильно менять уже внесенные измнения, но, возможно, нужно будет пересмотреть это).

На последнем звонке с командой решили, что уже будем приступать к реализации виджетов. Я с понедельника возьмусь за AI-чат, потому что трудно предсказать, сколько замет разработка.
